Accessing MongoDB With C#

Why Use MongoDB With C#?

MongoDB is a database designed to help apps read humongous amounts of data at top speed. Unlike relational (SQL) databases, MongoDB is a document-oriented database, meaning that all data is stored as BSON (Binary JSON) key-value pairs (where the values can be nested documents or arrays of nested documents). Because complex data structures can be stored as single documents, MongoDB doesn't have to perform any joins when reading data and can provide a much faster query response time.

Additionally, the documents in MongoDB collections (MongoDB has collections instead of tables) do not have to follow a rigidly-defined schema, so the data structures stored in MongoDB collections can be easily changed without the need for database migrations. In fact, documents in MongoDB collections can be store totally different data, if need be.

When your in-production app needs to scale for the increasing number of users, MongoDB makes it easy to scale horizontally with a technique called sharding. While sharding is possible with SQL databases (using entity-based sharding), the fact that data structures are spread out across many tables makes it a very complicated endeavor. Thus, SQL databases are usually scaled vertically, with a single server being upgraded to a better CPU and more RAM. But with document-oriented databases, few if any links between collections exist, which makes splitting the database into several shards, one on each cheaper server, a much simpler task.

Even though C# is known for great integrations with SQL databases, it has an excellent library for accessing MongoDB. The MongoDB.Driver package provides all the functionality necessary to develop awesome apps with MongoDB.

What drawbacks come from using a document-oriented database?

While MongoDB supports joins, it is not best suited for software that involves complicated graphs of relationships between objects (like an issue tracker where many organizations have many workers who collaborate on many projects, each of which has many workers and perhaps many sub-projects with many more workers). MongoDB shines in situations where the graph of a stored data structure doesn't have any cycles, like a tree or linked list.

Also, because updating data often requires changing deeply nested data, applications that have a high update rate, but do not need maximum read performance, may achieve better performance with a different paradigm of database design.

Setting Up a C# Project With MongoDB

To demonstrate adding MongoDB to a C# project, we'll update the C# web API from the Making a Web App with a C# Web API article. In the previous article, we created a C# web API that could generate random weather data and display it to a user, but it couldn't store the data permanently in a database. Once we add MongoDB to the app, the weather reports served to the front end will come from permanent records and not be randomly generated.

After either setting up an account with MongoDB Atlas or installing MongoDB locally (I'm using MongoDB locally), we'll create a new database with an initial collection named "Weather Forecasts" (all MongoDB databases are required to have at least one collection):

Next, in Visual Studio, right click on the project's Dependencies folder and select Manage NuGet Packages. In the NuGet Manager, browse for the MongoDB.Driver package and click Install. Now that this package is added to the project, we can create a new folder named 'DataAccess' and add a new class for data access called MongoDBAccessor to it:

using Microsoft.Extensions.Configuration; using MongoDB.Bson; using MongoDB.Driver; using System.Collections.Generic; using System.Linq; namespace WebAPIDemoForArticle.DataAccess { public class MongoDBAccessor { /// <summary> /// The link to the MongoDB database /// </summary> private readonly IMongoDatabase _database; private const string _forecastCollection = "Weather Forecasts"; public MongoDBAccessor(IConfiguration configuration) { MongoClient client = new MongoClient(); string database = configuration.GetConnectionString("MongoDB"); _database = client.GetDatabase(database); } /// <summary> /// Runs a query to get all the binary JSON (BSON) documents in the "Weather Forecasts" collection, /// using MongoDB's Find() method. /// </summary> public List<WeatherForecast> GetForecastsAsync() { IMongoCollection<WeatherForecast> collection = _database.GetCollection<WeatherForecast>(_forecastCollection); return collection.Find(new BsonDocument()).ToList(); } } }

You'll notice that the constructor of the MongoDBAccessor class takes an IConfiguration object, which is a type that represents the application's configuration settings and will be passed in by the ASP.NET Core dependency injection system. However, the configuration does not currently contain any connection strings for MongoDB. To add a connection string, you'll need to add a connection string to the appsettings.json file (because I'm using a local MongoDB instance, the connection string is simply the name of the database):

"ConnectionStrings": { "MongoDB": "DemoDB" },

Speaking of dependency injection, we'll need to add a new dependency to the project's startup code. In .NET 6.0 or later, you can add a new Singleton dependency in the Program.cs file:

builder.Services.AddControllers(); // Learn more about configuring Swagger/OpenAPI at https://aka.ms/aspnetcore/swashbuckle builder.Services.AddEndpointsApiExplorer(); builder.Services.AddSwaggerGen(); builder.Services.AddSingleton<MongoDBAccessor>();

In .NET 5.0 or earlier, you'll need to add a new Singleton in the Startup.cs file, at the end of Startup.ConfigureServices():

public void ConfigureServices(IServiceCollection services) { services.AddControllers(); services.AddSwaggerGen(c => { c.SwaggerDoc("v1", new OpenApiInfo { Title = "WebAPIDemoForArticle", Version = "v1" }); }); builder.Services.AddSingleton<MongoDBAccessor>(); }

The last step before changing the API controller to use the MongoDB accessor is to update the WeatherForecast class to be properly serialized and deserialized to and from BSON. Firstly, the class needs a Guid Id property that MongoDB will use as the unique identifier for each document. Next, the WeatherForecast class has a property called TemperatureF that simply converts the TemperatureC property to Fahrenheit. The TemperatureF property can be marked to be ignored when serializing by adding the [BsonIgnore] attribute to the property:

using MongoDB.Bson.Serialization.Attributes; using System; namespace WebAPIDemoForArticle { public class WeatherForecast { [BsonId] public Guid Id { get; set; } public DateTime Date { get; set; } public int TemperatureC { get; set; } [BsonIgnore] public int TemperatureF => 32 + (int)(TemperatureC / 0.5556); public string Summary { get; set; } } }
