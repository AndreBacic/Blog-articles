An Overview of Hypertext Transfer Protocol

What Is HTTP?

Hypertext Transfer Protocol (HTTP) is the underlying technology that allows web servers (like the ones Amazon and Google use to serve their websites) to respond to requests from web browsers (like you, with Chrome or Firefox or whichever browser you use). In other words, HTTP is the foundation for communication on the World Wide Web.

What Do HTTP Conversations Look Like?

First, a client sends a request to a web server. The web server then processes the request and sends a response back to the client, completing the HTTP exchange.

Both the request and the response contain important information in their headers, and often after the headers include a body. The request headers will tell the server if the client has any authentication credentials, and also includes metadata (information about information) about the body (if one is sent), and other relevant data about the client. If a request includes a body, it is typically data that the client wishes the server to store in a database.

The response headers will tell the client how the server processed the request, details about the server and connection, and metadata about the response body (if one is sent). The body of a response is the information the client requested, often a web page.

HTTP Methods

Let's now look at some specific HTTP information used in any request-response exchange.

Firstly, in a request, the client uses HTTP methods to ask a web server to perform specific actions.

The most common methods are:

  • GET : ask for data from the server.
  • POST : send data to the server for storage.
  • PUT : update data on the server.
  • DELETE : permanently remove data from the server.

The HTTP methods are case-sensitive, so a client cannot send a 'get' request to a web server and receive a successful response.

HTTP Status Codes

When the server replies with an HTTP response, the response will always include a status code, which tells the client how the server processed the request. Status codes are grouped into five categories, with the first digit representing the category:

  • 1xx : informational messages, such as "Continue" or "Switching Protocols".
  • 2xx : successful responses, such as "OK" or "Created".
  • 3xx : redirections, such as "Moved Permanently" or "See Other".
  • 4xx : client errors, such as "Bad Request" or "Unauthorized".
  • 5xx : server errors, such as "Internal Server Error" or "Not Found".

An Example HTTP Conversation

To put all this together, let's look at the HTTP request from your browser to get this article and the HTTP response from the my blog's web server; first looking at the request:

GET / HTTP/1.1 Host: andreblog.azurewebsites.net/article.html Connection: keep-alive User-Agent: Mozilla/5.0 (Windows; U; Windows NT 6.1; en-US) AppleWebKit/533.2 (KHTML, like Gecko) Chrome/6.0

After specifying the HTTP method and protocol version, the client sends in the headers the host name (url to the server and a specific webpage or API endpoint on it), connection type, and the user agent (the browser). The user agent shown here is actually Chrome, but it could be any browser. In POST or other such methods, the client sends the data in the body, which would follow below the headers. The headers would also include other metadata about the request, such as the content type of the body.

The server responds with:

HTTP/1.1 200 OK Content-Type: text/html Content-Length: 1163 Date: Wed, 06 Apr 2022 01:25:09 GMT Server: Kestrel <!DOCTYPE html> <html> <head> ...and the rest of the response body, which in this case is the webpage you're looking at.

The server sends a 200 OK response, which means the request was successful and includes the response body. The response headers include the content type and length of the response body, the type of server that sent the information, and the time the response was sent (the example given is when I was writing this article, but the response your computer received had the present time in the Greenwich Mean Time time zone).

Once your browser received the response, the HTTP conversation was over and your browser turned to applying CSS styles and running some JavaScript code, but that's information for another article.
